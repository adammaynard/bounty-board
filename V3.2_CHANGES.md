# Bounty Board v3.2 — Change Summary

## Overview

v3.2 introduces a clear conceptual separation between **in-app crypto (USDC)** and **out-of-band (OOB)** payments, plus three new admin/power-user features.

---

## Critical Change: OOB Payment Separation

### Architecture

| Concern | Mechanism | Notes |
|---|---|---|
| **Deposits** | USDC on Base L2 | User sends on-chain, submits tx hash, admin approves |
| **Withdrawals** | USDC on Base L2 | User provides Base L2 wallet address; auto-sent if hot wallet key is configured, otherwise queued for admin |
| **Quest payment (Platform)** | USDC escrow | Reward locked at claim time, released on approval |
| **Quest payment (OOB)** | Peer-to-peer (CashApp, Venmo, cash, etc.) | Both parties confirm; app has no visibility into actual transfer |

OOB methods (CashApp, Venmo, Zelle, etc.) are **quest-only** — they do not apply to fund requests.

### `cgi-bin/api.py` changes

1. **`handle_create_fund_request`** — Removed the `valid_methods` list (`['crypto', 'cashapp', 'venmo', 'zelle', 'other']`). The `method` field is now hardcoded to `'crypto'` regardless of what the client sends. The auto-withdraw trigger (`HOT_WALLET_PRIVATE_KEY` check) was simplified to not depend on `method == 'crypto'` since method is always crypto.

2. **`handle_admin_review_fund_request`** — Deposit transaction note now says `'Deposit via crypto (USDC)'` (previously used `method_label` which could be 'cashapp' etc.). Withdrawal note construction no longer appends `via {method}` since method is always crypto.

### `app.js` changes

1. **Deposit modal** (`deposit-btn` click handler):
   - Removed the Payment Method `<select>` dropdown (Crypto / CashApp / Venmo / Zelle / Other)
   - Transaction Hash is now **required** (previously optional)
   - Placeholder updated to `0x...` (previously `0x... or CashApp ref`)
   - Title updated to `Request Deposit (USDC)`
   - Description updated to clarify USDC on Base L2 only
   - Validation: now shows error if tx hash is empty

2. **Withdraw modal** (`withdraw-btn` click handler):
   - Removed the Payment Method `<select>` dropdown
   - Label updated to `Base L2 Wallet Address (required)` (previously `Wallet Address / Payment Info`)
   - Placeholder updated to `0x...` (previously `0x... or $CashTag or @VenmoHandle`)
   - Title updated to `Request Withdrawal (USDC)`
   - Description updated to clarify USDC on Base L2 only
   - `method` field removed from POST body (backend ignores it anyway)
   - Validation message updated to `Enter your Base L2 wallet address`

3. **User fund request list** (`renderFundRequests`):
   - Method badge now always displays `Crypto (USDC)` with class `badge-method-crypto`
   - Removed dynamic method badge that could show CashApp/Venmo/etc.

4. **Admin fund request list** (`renderAdminFundRequests`):
   - Same as user list — method badge always shows `Crypto (USDC)`
   - Removed `data-method` attribute from approve button (no longer needed)
   - Admin approve confirmation dialog: removed `reqMethod === 'crypto'` check; auto-transfer note is shown whenever `hotWalletInfo.auto_withdraw_enabled` is true
   - Manual approval prompt updated: `Manually send USDC on Base L2 to:` (previously generic `Make sure you have sent funds to:`)

5. **Quest creation form** — **Unchanged.** The Payment Method dropdown on the quest form (`Platform Balance` vs `Out of Band`) is a separate, unrelated concept and remains intact.

## Feature 1: Fully Automatic Crypto Withdrawals

**Status: Implemented** (by previous subagent, verified working)

- When a user submits a withdrawal request and the hot wallet private key is configured (`BOUNTY_HOT_WALLET_KEY` or `BOUNTY_HOT_WALLET_KEY_FILE` env var), the backend calls `send_usdc()` immediately
- On success: balance is deducted instantly, fund request is recorded as `approved` with the tx hash, response includes `auto_transferred: true` and `tx_hash`
- On failure: falls back to pending fund request for admin review; response includes `auto_transferred: false` and `fallback_reason`
- Frontend shows a "Transfer Complete" modal with tx hash and BaseScan link on auto-transfer
- Admin fund request approve flow also uses `send_usdc()` for manual-review crypto withdrawals when hot wallet is configured

**Configuration:**
```bash
export BOUNTY_HOT_WALLET_KEY="0xYourPrivateKeyHere"
# or
export BOUNTY_HOT_WALLET_KEY_FILE="/path/to/keyfile"
```

## Feature 2: Admin User Management

**Status: Implemented** (by previous subagent, verified working)

### Delete User (`/admin/users/delete`)
- Admin-only endpoint that permanently deletes a non-admin user
- Pre-deletion cleanup:
  1. Cancels all active quests posted by the user (status: posted/claimed/submitted/disputed)
  2. Creates `refund` transaction records for any platform-payment escrow
  3. Re-opens any quests the user was claiming (resets to `posted`)
  4. Deletes fund requests, transactions, quest changelog entries authored by user
  5. Deletes the user record
- Guards: cannot delete yourself; cannot delete another admin
- Frontend: "Delete" button appears in admin users table for non-admin, non-self users; confirmation modal shows destructive warning

### Allocate Balance (`/admin/users/adjust-balance`)
- Admin-only endpoint accepts `target_user_id`, `amount` (positive or negative), `reason`
- Creates `admin_credit` transaction for positive amounts, `admin_debit` for negative
- Frontend: "Adjust Balance" button in admin users table; modal with amount + reason fields

## Feature 3: Full Quest Editing with Changelog

**Status: Implemented** (by previous subagent, verified working)

### Admin Edit Quest (`/admin/quests/edit`)
- Edits all quest fields: title, description, category, difficulty, min_reward, max_reward, escalation_type, escalation_period_hours, status, payment_method
- Changelog: every changed field is recorded in `quest_changelog` table with old value, new value, who changed it, and when
- Financial side-effects on edit:
  - If status changes to `cancelled` for an active platform quest → creates escrow refund transaction
  - If reward amounts change on a claimed/submitted/disputed quest → clamps `current_locked_reward` to new max_reward

### Quest Changelog (`/admin/quests/changelog`)
- Admin-only GET endpoint returns ordered history of all field changes for a quest
- Frontend: "Changelog" button on each quest in admin panel; displays a scrollable modal with field name, old→new values, changer's username, and timestamp

### `quest_changelog` table schema
```sql
CREATE TABLE IF NOT EXISTS quest_changelog (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    quest_id INTEGER NOT NULL,
    changed_by INTEGER NOT NULL,
    field_name TEXT NOT NULL,
    old_value TEXT,
    new_value TEXT,
    changed_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (quest_id) REFERENCES quests(id),
    FOREIGN KEY (changed_by) REFERENCES users(id)
);
```
